# 한국어 기술 블로그 스타일 가이드

## 톤 매핑 (내용 → 톤)

| 내용 유형 | 톤 |
|----------|-----|
| 트러블슈팅 (버그/장애) | 스토리텔링 — 문제 발견 → 추측 → 시행착오 → 해결, 시간 순 |
| 신기술 도입 후기 | 비교·평가 — 왜 선택, 안 좋은 점, 좋은 점 |
| 튜토리얼 (How-to) | 단계별 — 1, 2, 3 명확히, 각 단계마다 결과 확인 |
| 성능 최적화 | Before/After — 측정값으로 시작, 측정값으로 끝 |
| 회고/돌아보기 | 내러티브 — 시점 명확, 감정/판단도 OK |
| 라이브러리 만들기 | 동기 → 설계 → 트레이드오프 → 사용 예 |

## PAAR 프레임 (블로그 vs 이력서)

같은 작업을 두 산출물로 쓸 때 톤이 다름:

- **블로그 본문**: 스토리텔링 OK — 문제 발견 → 시행착오 → 해결 → 회고 순서 권장
- **이력서 항목**: **PAAR 순서 엄격히** — Problem → Action → (Analysis) → Result

블로그에서도 **결론/마무리 단락에 정량 Result 1줄은 반드시 포함**:
- ✅ "쿼리 응답이 평균 420ms → 68ms로 떨어졌습니다"
- ❌ "훨씬 빨라졌습니다" (정량성 없음)

Analysis(배운 점, 트레이드오프)는 블로그에서는 회고/마무리에 녹여도 되고, 이력서에서는 핵심 bullet 하나로 분리.

## 글 구조 (중간 길이 ~3000자)

1. **제목** — 구체적이고 검색되는 키워드 포함
   - ❌ "Flutter 개발기" (모호)
   - ✅ "Flutter Riverpod에서 Stream 메모리 누수 잡은 이야기"
   - ✅ "Isolate로 이미지 100장 처리 시간 5초 → 0.8초"

2. **도입 (200-300자)**
   - 첫 문장은 상황 묘사로 시작 (✅ "어느 날 앱이 갑자기 느려졌습니다")
   - ❌ 절대 안 됨: "안녕하세요, 오늘은 ~에 대해 알아보겠습니다"
   - 이 글 읽으면 뭘 얻는지 1줄로 명시

3. **본문 (소제목 2-4개)**
   - 소제목은 `##` 사용, 의문문 형태도 OK ("왜 이게 안 됐을까?")
   - 코드 블록 전후로 1-2줄 설명 (코드만 던지지 않기)
   - 긴 코드는 핵심만 발췌, `// ... 생략` 표시

4. **코드 스니펫 규칙**
   - 언어 명시 필수: ` ```dart `, ` ```typescript `, ` ```kotlin `
   - 변수/함수명은 영어 (한국어 안 섞기)
   - 코드 내 주석은 한국어 OK

5. **회고/마무리 (200-300자)**
   - "배운 점", "다음에 시도할 것" 같은 명시적 마무리 OK
   - 또는 1-2줄 핵심 요약으로 끝

## 회피할 AI 티 표현

- "~을 통해 알아보았습니다"
- "이번 포스팅에서는"
- "다양한 방법이 있습니다" (구체성 없음)
- "쉽고 간단하게" (안 쉬움)
- 이모지 남발 (제목에 1개 정도가 한계)
- "마치며" → "끝으로", "정리하면", "한 줄로" 등으로 변주

## 좋은 시작 문장 예시

- "지난 주에 [구체적 상황]"
- "[기술 X]를 도입했더니 [예상 못한 일]"
- "[숫자]개의 [무엇]을 처리해야 했습니다"
- "코드 리뷰에서 ~ 지적을 받았습니다"
- "출시 직후 사용자 제보가 들어왔습니다"

## 길이 가이드

- 짧음: 1500자 (단순 팁, 기록)
- **중간 (디폴트): 2500-3500자**
- 김: 5000자+ (튜토리얼, 깊은 분석)

기본 디폴트는 중간. 사용자 요청 있으면 조정.
